# 重复文件回收站设计

## 目标

将重复文件清理从不可恢复的直接删除改为可恢复的项目内回收站，并在文件进入回收站满 7 天后自动清理。

## 范围

- 仅替换“检测重复文件”结果中的清理操作。
- 既有重命名日志与其回滚功能不变。
- 不调用操作系统回收站，保持 Windows、macOS 和 Linux 上的一致行为。

## 方案选择

采用项目内托管回收站，而不是系统回收站或复制备份：

- 文件通过 `rename` 移入 `storage/trash/<id>/`，避免额外复制和空间占用。
- 每个条目写入 JSON 元数据，记录原始完整路径、回收时间和存储位置。
- 回收站页面可列出条目并恢复；恢复目标已存在时不覆盖，返回错误。
- 自动清理是机会式的：每次列出、移入或恢复回收站时，先删除已超过 7 天的条目。应用未运行时不启动后台任务；下一次相关操作会完成清理。

## 数据与目录

```
storage/
  trash/
    <id>/
      file              # 被回收的原始文件
      metadata.json     # id、originalPath、originalName、trashedAt、expiresAt
```

`id` 由当前时间和随机字节组成，避免同名文件冲突。回收的文件保留原文件名；元数据用于恢复。

## 后端接口

新增 `TrashManager`，职责是移入、列出、恢复和清理过期条目。

- `POST /api/duplicates/clean`：保留现有请求格式 `{ path, files }`，改为移动到回收站；成功状态为 `trashed`。
- `POST /api/trash`：返回未过期的回收站条目，并在返回前清理过期条目。
- `POST /api/trash/restore`：请求 `{ id }`，将文件恢复到元数据的 `originalPath`；目标存在时返回错误，绝不覆盖。

所有回收站操作都返回 JSON；单项失败不影响其他项。

## 前端交互

- 重复文件操作按钮改为“移入回收站”，确认提示说明可在 7 天内恢复。
- 新增回收站面板，显示原文件名、原始路径、回收时间、剩余天数和“恢复”按钮。
- 每次完成移动或恢复后刷新回收站与重复文件检测结果。

## 安全与错误处理

- 恢复只使用服务端记录的 `originalPath`，不接受客户端提供的任意恢复路径。
- 文件移动前验证源文件存在；恢复前验证回收文件和元数据存在。
- 清理过期条目只删除 `storage/trash` 内、由元数据标记为已过期的条目。
- 回收站不可写、跨卷移动失败或元数据损坏时，返回具体错误，源文件不做直接删除。

## 测试

使用独立 PHP 测试脚本覆盖：

1. 文件移入回收站后，原路径不存在，回收文件与元数据存在。
2. 恢复后，文件回到原路径，回收条目移除。
3. 原路径已有同名文件时，恢复失败且回收文件保留。
4. 已超过 7 天的条目被清理，未过期条目保留。
